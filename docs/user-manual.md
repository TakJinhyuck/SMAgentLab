## 1. 시작하기

### 접속

| 서비스 | URL |
|--------|-----|
| Ops-Navigator UI | http://localhost:8501 |
| API 문서 (Swagger) | http://localhost:8000/docs |

### 실행 방법

```bash
# Ollama 시작 (최초 1회 또는 재시작 시)
ollama serve

# 전체 스택 실행
docker compose up -d

# 상태 확인
docker compose ps
```

### 종료 방법

```bash
docker compose down
```

## 2. 시스템이 검색하는 방식 이해하기

검색을 잘 활용하려면 내부 동작을 간단히 알아두면 좋다.

### 벡터 검색 (의미 기반)

텍스트를 768개의 숫자로 변환해 의미가 비슷한 문서를 찾는다.

```
"쿠폰 강제 반납"  ──[임베딩 모델]──▶  [0.12, -0.34, 0.87, ...]
"쿠폰 회수 처리"  ──[임베딩 모델]──▶  [0.11, -0.31, 0.89, ...]

두 벡터의 거리 = 매우 가까움 → 유사한 문서로 판단
```

**장점:** 단어가 다르더라도 의미가 같으면 찾아준다.
**약점:** 특정 코드명, 시스템명 등 정확한 단어 매치에는 약하다.

### 키워드 검색 (단어 기반)

문서에서 정확한 단어 빈도를 계산해 매치하는 방식 (PostgreSQL FTS).

**장점:** `COUPON_001`, `coupon_issue` 같은 정확한 단어 매치에 강하다.
**약점:** 단어 표현이 조금만 달라도 못 찾는다.

### 하이브리드 = 두 방식 결합

```
최종 점수 = (벡터 비중 × 벡터 점수) + (키워드 비중 × 키워드 점수)
           × (1 + 문서 가중치)
```

두 방식의 단점을 서로 보완한다. 사이드바 슬라이더로 비중을 조절할 수 있다.

### 용어집의 역할

```
사용자 질문에 등장하는 모호한 표현을 표준 용어로 자동 변환한다.

"뺏어오기" ──[용어집]──▶ "회수"
→ 검색어에 "회수"가 추가되어 키워드 매치 정확도 향상
→ 결과 카드에 "🔤 용어 매핑: 회수" 표시
```

**용어집 없이 검색:** 사용자가 입력한 단어 그대로 검색
**용어집 있을 때:** 표준 용어로 강화된 쿼리로 검색 → 더 정확한 결과

## 3. 운영 보조 챗 사용법

좌측 메뉴에서 **💬 Chat** 페이지를 선택한다.

### 화면 구성

```
┌─────────────────────────────────────────────────────────────┐
│  사이드바                    │  채팅 영역                    │
│                              │                              │
│  ▼ coupon   (Namespace)      │  [현재 대화 제목]             │
│  ──────────────────          │  ─────────────────────────   │
│  ✏️ 새 대화                  │  [사용자] 질문 내용           │
│                              │                              │
│  최근                        │  [AI] 🔤 용어 매핑: 회수      │
│  쿠폰 회수 방법  ···          │                              │
│  주문 상태 조회  ···          │  🐳 컨테이너  🗂️ 관련 테이블  │
│  (···클릭 → 삭제)            │  coupon-api  coupon_issue    │
│  ──────────────────          │                              │
│  ⚙️ 검색 설정 ▶ (접기/펼치기)│  📋 SQL 쿼리                  │
│   벡터 0.7 / 키워드 0.3      │  SELECT * FROM ...           │
│   결과 수 Top-5              │                              │
│                              │  🤖 AI 답변                   │
│                              │  쿠폰 회수 처리는...          │
│                              │                              │
│                              │  👍  👎                       │
│                              │                              │
│                              │  [운영 질문을 입력하세요...]  │
└─────────────────────────────────────────────────────────────┘
```

### 질문하기

1. 하단 **채팅 입력창**에 질문을 자연어로 입력한다.
2. Enter 또는 전송 버튼을 누른다.
3. 결과가 표시될 때까지 잠시 기다린다.

**질문 예시:**
- "쿠폰 회수 실패한 건 어떻게 확인해?"
- "선물 발송 오류 로그 어디서 봐?"
- "주문 상태가 결제완료인데 배송 안 되는 케이스 쿼리 알려줘"

### 결과 카드 읽기

| 항목 | 설명 |
|------|------|
| 🔤 용어 매핑 | 입력한 표현이 표준 용어로 변환된 경우 표시 (예: "뺏어오기" → "회수") |
| 🐳 컨테이너 | 해당 처리를 담당하는 서버/서비스명 |
| 🗂️ 관련 테이블 | 조회해야 할 DB 테이블 목록 |
| 📋 SQL 쿼리 | 바로 실행 가능한 SQL 템플릿 |
| 📄 검색된 문서 | 관련도 순 전체 문서 목록 (펼치기 가능) |
| 🤖 AI 답변 | LLM이 참고 문서를 기반으로 생성한 답변 |

### 피드백 남기기

답변 하단의 **👍 / 👎** 버튼으로 유용도를 평가한다. 피드백은 단순 로그가 아니라 **시스템 품질을 직접 개선**한다.

| 버튼 | 즉각 효과 | 누적 효과 |
|------|----------|----------|
| 👍 좋아요 | 해당 문서 `base_weight` +0.1 → 다음 유사 질문에서 이 문서가 더 높이 검색됨 | 이번 Q&A가 "사례 DB"에 저장 → 유사 질문 시 LLM에게 **참고 답변 예시**로 제공됨 (few-shot) |
| 👎 싫어요 | 해당 문서 `base_weight` -0.1 → 검색 랭킹 하락 | 미해결 케이스로 분류 → Admin 통계에서 확인 가능, 지식 보완 필요 신호 |

**👎 클릭 후 지식 즉시 개선:**

👎를 누르면 아래에 선택적 **지식 베이스 개선 폼**이 표시된다.
올바른 가이드 내용을 작성해 등록하거나, **건너뛰기**로 생략할 수 있다.
등록한 지식은 `base_weight=1.5`(기본값)로 높게 설정되어 다음 유사 질문에서 우선 검색된다.

> **피드백을 많이 남길수록 AI 답변이 좋아집니다.** 특히 👍는 실제 검색 품질과 LLM 답변 수준을 동시에 개선합니다.

> 피드백은 1회 제출 후 변경 불가합니다.

### 대화 목록 관리

- 사이드바 상단 **✏️ 새 대화** 버튼으로 새 대화를 시작한다.
- 과거 대화는 **최근** 섹션에 목록으로 표시된다. 클릭하면 이전 대화를 불러온다.
- 각 대화 항목 우측 **`···`** 버튼 클릭 → **🗑️ 삭제 / 취소** 버튼이 인라인으로 나타난다.

### 사이드바 설정

| 설정 | 기본값 | 설명 |
|------|--------|------|
| Namespace | - | 검색할 도메인 선택 (coupon, gift, order 등) |
| ⚙️ 검색 설정 | (접힘) | 클릭하면 펼쳐지며 아래 슬라이더가 노출됨 |
| 벡터 비중 | 0.7 | 의미 기반 유사도 반영 비율 |
| 키워드 비중 | 0.3 | 단어 일치 반영 비율 (자동 = 1 - 벡터 비중) |
| 결과 수 (Top-K) | 5 | 검색 결과 최대 개수 |

## 4. 네임스페이스 관리

좌측 메뉴에서 **🗄️ Admin** > **🏷️ 네임스페이스** 탭을 선택한다.

Namespace는 운영 도메인을 구분하는 가장 큰 단위다. 지식 베이스와 용어집이 namespace 단위로 완전히 분리되므로, 도메인 간 검색 간섭이 없다.

### Namespace 추가

1. 하단 **새 Namespace 추가** 폼에 이름과 설명을 입력한다.
2. **추가** 버튼을 클릭한다.

> 이름은 영문 소문자·숫자·하이픈 사용 권장 (예: `coupon`, `order`, `delivery`)

### Namespace 삭제

1. 목록에서 삭제할 namespace 우측 **🗑️ 삭제** 버튼을 클릭한다.
2. 등록된 지식/용어집 건수가 표시되며 삭제 여부를 확인한다.
3. **✅ 삭제 확인** 버튼을 눌러야 실제 삭제된다.

> ⚠️ Namespace 삭제 시 해당 namespace의 **지식, 용어집, 피드백, 질의 로그가 모두 영구 삭제**됩니다.

## 5. 지식 베이스 관리

좌측 메뉴에서 **🗄️ Admin** > **📚 지식 베이스** 탭을 선택한다.

### 지식 등록

1. **➕ 신규 지식 등록** 섹션을 펼친다.
2. 아래 항목을 입력한다.

| 필드 | 필수 | 설명 |
|------|------|------|
| Namespace | ✅ | 드롭다운 선택 (네임스페이스 탭에서 먼저 추가 필요) |
| 컨테이너명 | | 해당 서비스의 컨테이너명 (예: `coupon-api`) |
| 관련 테이블 | | 쉼표로 구분 (예: `coupon_issue, coupon_use_history`) |
| 가이드 내용 | ✅ | 운영 처리 방법, 원인 설명 등 상세 기술 |
| SQL 쿼리 템플릿 | | 직접 실행 가능한 SQL문 |

3. **등록** 버튼을 클릭한다.
4. 등록 즉시 임베딩이 자동 생성되어 검색에 반영된다.

**좋은 가이드 내용 작성 예시:**
```
쿠폰 강제 회수 처리

발생 상황: 쿠폰이 부정 사용 또는 시스템 오류로 잘못 발급된 경우
처리 방법:
1. coupon_issue 테이블에서 해당 쿠폰 ID 확인
2. coupon_use_history에서 사용 여부 확인
3. 미사용 상태면 coupon_issue.status = 'RECALLED' 업데이트
4. 사용된 경우 CS팀에 에스컬레이션

관련 시스템: coupon-api, promotion-batch
담당팀: 쿠폰플랫폼팀
```

### 지식 수정

1. 목록에서 해당 항목의 **expander**를 펼친다.
2. **✏️ 수정** 버튼을 클릭한다.
3. 수정할 필드를 변경 후 **저장**을 클릭한다.
4. 내용(content)이 변경된 경우 임베딩이 자동으로 재생성된다.

### 지식 삭제

1. 해당 항목 expander를 펼친다.
2. **🗑️ 삭제** 버튼을 클릭한다.

> 삭제는 즉시 처리되며 복구할 수 없습니다.

## 6. 용어집 관리 — 검색 정확도의 핵심

**🗄️ Admin** > **📖 용어집** 탭을 선택한다.

용어집은 **검색 정확도를 높이는 핵심 기능**이다.
사용자가 어떤 표현을 써도 시스템이 올바른 표준 용어로 연결해 준다.

### 등록 전략

**가장 효과적인 용어집 구성 방법:**

1. **운영팀 내 은어·줄임말 수집**
   - 정식 명칭: "쿠폰 회수" → 팀 은어: "뺏어오기", "강취", "강제 취소"

2. **설명 필드를 풍부하게 작성** (이것이 실제 임베딩에 사용됨)
   ```
   설명: 쿠폰 회수, 쿠폰 뺏어오기, 강제 반납, 강제 취소, 발급 취소,
        잘못된 쿠폰 무효화, 부정 사용 쿠폰 회수, 쿠폰 강제 만료 처리
   ```

3. **Namespace별로 분리 관리**
   - coupon namespace: 쿠폰 관련 용어
   - gift namespace: 선물 관련 용어
   - order namespace: 주문 관련 용어

### 용어 등록이 검색에 미치는 영향

```
Before (용어집 없음):
  질문: "쿠폰 강취 확인"
  검색: "쿠폰 강취 확인" 그대로 → "강취" 단어 없는 문서 미매치

After (용어집 있음):
  질문: "쿠폰 강취 확인"
  매핑: "강취" → "회수" (벡터 유사도)
  검색: "쿠폰 강취 확인 회수" → "회수" 포함 문서 키워드 매치 성공
```

### 용어 등록

| 필드 | 필수 | 설명 |
|------|------|------|
| Namespace | ✅ | 드롭다운 선택 (네임스페이스 탭에서 먼저 추가 필요) |
| 표준 용어 | ✅ | 시스템에서 사용하는 공식 표현 |
| 설명 및 유의어 | ✅ | 사용자가 쓸 수 있는 다양한 표현과 맥락 설명 |

**예시:**
```
Namespace:  coupon
표준 용어:  회수
설명 및 유의어:
  쿠폰 회수, 쿠폰 뺏어오기, 강제 반납, 회수 처리, 쿠폰 취소,
  발급 취소, 잘못된 쿠폰 제거 등 쿠폰을 강제로 무효화하는 행위
```

> 설명이 상세할수록 유사 표현 매핑 정확도가 높아집니다.

### 용어 수정

1. 목록에서 해당 항목 expander를 펼친다.
2. **✏️ 수정** 버튼을 클릭한다.
3. 표준 용어 또는 설명을 수정 후 **저장**을 클릭한다.
4. 설명이 변경된 경우 임베딩이 자동으로 재생성된다.

### 용어 삭제

1. 해당 항목 expander를 펼친다.
2. **🗑️ 삭제** 버튼을 클릭한다.

## 7. 통계 대시보드

**🗄️ Admin** > **📊 통계 대시보드** 탭을 선택한다.

시스템이 얼마나 잘 동작하고 있는지 한눈에 파악하고, 어디를 보완해야 할지 알 수 있다.

### 상단 KPI 카드

| 지표 | 설명 | 목표 |
|------|------|------|
| 💬 전체 질의 수 | 누적 질문 수 | — |
| ✅ 전체 해결률 | AI가 답변을 찾은 비율 | 80% 이상 |
| 👍 사용자 만족도 | 좋아요 / 전체 피드백 비율 | 70% 이상 |
| 🚨 미해결 케이스 | 답을 못 찾은 질문 수 | 0건 목표 |

### 지식 베이스 현황

namespace별 지식 문서 수와 용어집 수를 테이블로 보여준다.
**지식 수가 0인 namespace는 노란색으로 강조** — 해당 namespace에 지식을 먼저 등록해야 한다.

### 질의 처리 현황

해결된 질문(초록)과 미해결 질문(빨강)을 namespace별 스택 바차트로 표시한다.
빨강 영역이 크면 → 해당 namespace의 지식 베이스를 보완해야 한다.

### 사용자 피드백

👍/👎 비율 차트와 namespace별 만족도 텍스트 게이지(`🟢/🟡/🔴`)를 함께 보여준다.
- 🟢 70% 이상: 양호
- 🟡 40~70%: 지식 품질 점검 필요
- 🔴 40% 미만: 즉시 지식 개선 필요

### 미해결 케이스 — 지식 보완 가이드

페이지 하단에 **🚨 미해결 케이스** 목록이 표시된다.

> "아래 질문들에 AI가 적절한 답변을 찾지 못했습니다."

1. 미해결 질문 목록에서 **📝 등록** 버튼을 클릭한다.
2. 해당 질문 내용이 미리 채워진 **인라인 지식 등록 폼**이 나타난다.
3. 가이드 내용과 SQL을 작성해 **✅ 등록 & 미해결 제거** 버튼을 클릭한다.
4. 지식이 등록됨과 동시에 해당 미해결 로그가 자동 삭제된다.

미해결 케이스 발생 조건:
- 검색 결과가 없는 질문 (관련 지식 없음)
- 사용자가 👎 를 누른 질문 (검색은 됐지만 내용이 틀림)

### 새로고침

**🔄 새로고침** 버튼을 클릭하면 최신 데이터로 갱신된다.

## 8. 벡터 검색 테스트 (Admin 전용)

**🗄️ Admin** > **🔬 벡터 검색 테스트** 탭을 선택한다.

LLM 없이 검색 파이프라인 전 과정을 즉시 확인할 수 있는 디버그 도구다.
지식을 새로 등록했거나 용어집을 수정했을 때, Chat까지 가지 않고 바로 영향을 검증할 수 있다.

### 사용 방법

1. Namespace와 질문을 입력하고 **🔍 검색 실행**을 클릭한다.
2. 아래 4개 섹션에서 결과를 확인한다.

### 결과 섹션 설명

**Step 1 — 용어집 매핑**

| 항목 | 설명 |
|------|------|
| 매핑된 표준 용어 | 질문과 가장 유사한 용어집 항목의 `term` |
| 유사도 | 0~1 사이 값. 높을수록 질문과 용어집 설명이 의미적으로 가깝다 |
| 용어집 설명 | 임베딩에 사용된 원본 설명 텍스트 |
| 강화된 검색어 | 원본 질문 + 매핑된 용어 = 실제 검색에 사용된 쿼리 |

> 용어 매핑이 없음으로 나오면 → 해당 namespace에 용어집이 없거나 유사한 용어가 없는 것.
> Admin > 용어집에서 관련 표현을 등록하면 된다.

**Step 2 — 검색 파라미터**

현재 적용된 w_vector / w_keyword / 반환 건수 확인.

**Step 3 — 검색 결과 및 점수**

각 문서에 대해 다음을 표시한다:

| 지표 | 의미 |
|------|------|
| 🔵 벡터 점수 (v_score) | 코사인 유사도 (0~1). 질문과 문서 내용의 의미적 유사도 |
| 🟡 키워드 점수 (k_score) | ts_rank 점수. enriched_query 단어가 문서에 얼마나 매치됐는지 |
| ⚖️ base_weight | 해당 문서의 수동 가중치 |
| 🏆 최종 점수 | `(w_vec × v_score + w_kw × k_score) × (1 + base_weight)` |

막대 차트로 벡터 기여 vs 키워드 기여 비율을 시각적으로 비교할 수 있다.

**Step 4 — LLM 컨텍스트 미리보기**

실제 Chat 호출 시 LLM 프롬프트에 삽입될 텍스트를 미리 볼 수 있다.
내용이 부실하거나 잘못된 문서가 포함됐다면 해당 지식을 수정한다.

### 활용 시나리오

```
1. 새 지식 등록 후 검색 결과 확인
   → 등록한 문서가 상위에 나타나는지 검증

2. 용어집 추가 후 매핑 확인
   → "매핑된 표준 용어"에 의도한 용어가 나오는지, 유사도가 충분한지 확인

3. 검색 결과가 Chat에서 엉뚱하게 나올 때
   → v_score/k_score 중 어느 쪽이 낮은지 파악
   → 낮은 쪽 비중을 줄이거나 (Chat 슬라이더), 용어집/지식 내용 보완

4. base_weight 조정 효과 검증
   → 가중치 변경 후 최종 점수 변화 즉시 확인
```

## 9. 검색 품질 튜닝

### 벡터 vs 키워드 비중 조절

| 상황 | 권장 설정 |
|------|----------|
| 개념적 질문 (의미 중심) | 벡터 0.8 이상 |
| 특정 용어/코드 포함 질문 | 키워드 0.5 이상 |
| 일반적인 운영 질문 | 기본값 (벡터 0.7 / 키워드 0.3) |

**예시:**
- "장애 대응 절차가 어떻게 돼?" → 벡터 0.9 (의미 유사성 중요)
- "COUPON_001 에러 코드 원인은?" → 키워드 0.6 (정확한 단어 매치 중요)

### Top-K 조절

- 결과가 너무 많으면 Top-K를 3으로 줄인다.
- 원하는 문서가 안 나오면 Top-K를 10으로 늘린다.

### 지식 등록 팁

**내용을 풍부하게 작성한다:**
- 발생 상황, 원인, 처리 방법을 모두 포함
- 담당 시스템, 테이블, 관련 코드명 명시
- 실제 운영자가 쓰는 표현 포함 (구어체 OK)

**SQL 쿼리 템플릿 활용:**
- 바로 복사해서 실행 가능한 형태로 작성
- `WHERE id = :target_id` 형식으로 변수 표시

**용어집을 먼저 구축한다:**
- 팀 내 은어, 줄임말, 시스템 특유 표현을 등록
- 용어집이 풍부할수록 검색 정확도 향상

## 10. 자주 묻는 질문 (FAQ)

**Q. 검색 결과가 아예 안 나와요.**
- 해당 Namespace에 지식이 등록되어 있는지 Admin에서 확인하세요.
- 지식이 있다면 용어집에 관련 표현을 등록해 보세요.

**Q. AI 답변이 "[LLM 서버에 연결할 수 없습니다...]"로 나와요.**
- Ollama가 실행 중인지 확인하세요: `curl http://localhost:11434/api/tags`
- 중단된 경우 `ollama serve`로 재시작하세요.
- AI 답변 없이도 검색 결과 카드와 SQL은 정상 표시됩니다.

**Q. AI 답변이 너무 오래 걸려요 (timeout).**
- CPU 환경에서 7.8B 모델은 응답 생성에 수 분이 소요될 수 있습니다.
- 현재 타임아웃은 **900초(15분)** 로 설정되어 있습니다.
- 더 빠른 응답이 필요하면 더 작은 모델로 교체하세요:
  ```bash
  # 가벼운 모델 옵션 (PoC용)
  ollama pull exaone3.5:2.4b     # exaone 경량 버전
  ollama pull qwen2.5:3b         # 빠르고 한국어 준수
  ollama pull llama3.2:3b        # Meta 3B 모델
  ```
  `.env` 파일에서 `OLLAMA_MODEL=qwen2.5:3b` 변경 후 `docker compose up -d backend`
- 참고: 검색 결과(컨테이너/테이블/SQL)는 LLM과 무관하게 즉시 표시됩니다.

**Q. 같은 질문인데 매번 다른 결과가 나와요.**
- 벡터/키워드 비중 슬라이더가 변경되지 않았는지 확인하세요.
- LLM 답변은 생성 모델 특성상 매번 다를 수 있습니다.
- 검색 결과 카드(SQL, 컨테이너 등)는 동일한 결과가 유지됩니다.

**Q. 지식을 등록했는데 검색에 반영이 안 돼요.**
- 임베딩 생성에 수 초가 소요됩니다. 잠시 후 다시 검색해보세요.
- Backend 로그를 확인하세요: `docker compose logs backend --tail=50`

**Q. 새 Namespace를 추가하려면?**
- 별도 설정 없이 지식 등록 시 Namespace 필드에 새 이름을 입력하면 자동 생성됩니다.

**Q. 사내 LLM으로 교체하고 싶어요.**
- `.env` 파일에서 아래 값을 설정 후 `docker compose up -d backend`로 재시작하세요:
  ```
  LLM_PROVIDER=inhouse
  INHOUSE_LLM_URL=http://your-llm-gateway/v1
  INHOUSE_LLM_API_KEY=your-api-key
  INHOUSE_LLM_MODEL=your-model-name
  ```

**Q. 데이터 백업은 어떻게 하나요?**
```bash
# DB 백업
docker exec ops-postgres pg_dump -U ops opsdb > backup.sql

# DB 복원
docker exec -i ops-postgres psql -U ops opsdb < backup.sql
```
